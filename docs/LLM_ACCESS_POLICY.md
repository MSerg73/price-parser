# Политика обращения к LLM

- Версия политики: `1.1`
- Действует с: `0.7.1rc3`
- Статус: **обязательное правило**

## 1. Роль LLM

LLM помогает разбирать спорные строки, но не является источником истины.

Приоритет источников:

1. исходный файл и координаты;
2. детерминированный парсер;
3. подтверждённые бизнес-правила;
4. программная валидация;
5. решение оператора.

## 2. Когда разрешён вызов

Только для явно выбранных строк с причиной REVIEW:

- конфликт марки;
- неоднозначный профиль;
- неполный размер;
- низкая уверенность детерминированной проверки;
- другой зарегистрированный код причины.

Передача полного прайса запрещена.

## 3. Условия live

- явный провайдер;
- `--confirm-live-api`;
- модель задана явно или через `LLM_MODEL`;
- проверенный `batch_size=1`, допустимый диапазон `1–10`;
- пилот ограничен 25 строками;
- `source_id` должен сохраниться;
- gold и `case_type` не передаются;
- Structured Output;
- web/tools не подключаются автоматически;
- supplier/description считаются недоверенными данными.

## 4. API-ключ

Основная переменная: `OPENAI_API_KEY`.

`LLM_API_KEY` поддерживается только для legacy-совместимости. Если обе переменные
заданы и различаются, выполнение останавливается.

Ключи запрещено:

- помещать в код;
- сохранять в evidence;
- выводить в консоль;
- включать в delivery ZIP или Git.

## 5. Retry

Повторяются только временные ошибки:

- timeout/connection;
- HTTP 408;
- HTTP 409;
- HTTP 429;
- HTTP 5xx.

HTTP 400/401/403/404 автоматически не повторяются.

## 6. Результат

Результат LLM сохраняется как предложение для REVIEW:

- исходные значения;
- предложенные значения;
- confidence;
- warnings;
- модель;
- токены;
- latency;
- стоимость;
- fingerprint;
- replay.

`automatic_application_performed=false` обязательно.

## 7. Replay/reconciliation

- raw response сохраняется до постобработки;
- replay не вызывает API;
- live и replay проходят одинаковую идемпотентную reconciliation;
- сравнение выполняется после одинаковой обработки.

## 8. Приёмка

- 15/15 control оценены;
- fully correct не менее 90%;
- каждое поле не менее 95%;
- source_id и схема валидны в 100%;
- токены, latency и стоимость записаны;
- replay совпадает с live;
- автоприменение отсутствует.

Фактический результат `0.7.1rc3`: все критерии пройдены.

## 8.1. Условный вызов

- в LLM передаются только строки, выбранные `collect_candidates`;
- если REVIEW-кандидатов нет, вызов не выполняется;
- статус такого шага: `NOT_NEEDED`;
- токены и оценочная стоимость равны нулю;
- обычные строки не используются как fallback для заполнения пакета.

## 9. КРУГ/ПРУТОК

- LLM сохраняет исходное название профиля в нормализованной строке;
- `КРУГ` не переписывается в `ПРУТОК` и наоборот;
- для обоих `dim1` — диаметр;
- `dim2` — явно указанный второй размер, обычно длина;
- отсутствующий `dim2` не выдумывается;
- поисковая эквивалентность `КРУГ/ПРУТОК` применяется программно после
  нормализации и не зависит от решения модели.
