# Архитектура поиска номенклатуры v0.3.0

## Назначение

Модуль `price_parser.nomenclature` выполняет локальный объяснимый поиск по
нормализованным позициям поставщиков. Он не вызывает LLM и не применяет
найденные замены автоматически.

## Поток

```text
SearchQuery
→ нормализация профиля, марки и размеров
→ точное совпадение
→ подтверждённый псевдоним
→ подтверждённая связь марок
→ предложенная связь марок
→ fuzzy-кандидат
→ проверка размеров
→ сортировка и объяснение
→ SearchResponse
```

## Компоненты

- `models.py` — типизированные модели запроса, справочника и результата;
- `normalization.py` — безопасные ключи марки и разбор до трёх размеров;
- `catalog.py` — загрузка и валидация версионируемого справочника;
- `repository.py` — загрузка JSONL наличия и адаптер `ParsedItem`;
- `scoring.py` — объяснимая оценка строкового и размерного сходства;
- `service.py` — последовательность уровней поиска;
- `errors.py` — доменные ошибки.

Старый `price_parser.search.search_items()` сохранён без изменения для обратной
совместимости с экспортом v0.2.6.

## Приоритеты

1. Совпадение нормализованного написания марки.
2. Подтверждённый псевдоним той же канонической марки.
3. Подтверждённая связь марок.
4. Предложенная, но не подтверждённая связь.
5. Fuzzy-кандидат.
6. Близкий размер.

Точный профиль обязателен. Межпрофильные результаты не возвращаются.

## Безопасность решений

- `automatic_application_performed` всегда равно `false`;
- proposed-эквивалент всегда получает `requires_review=true`;
- fuzzy никогда не подтверждает эквивалентность;
- близкий размер всегда требует подтверждения менеджера;
- подтверждённая запись справочника обязана иметь источник;
- повреждённый или противоречивый справочник останавливает поиск понятной ошибкой.

## Расширение

Справочник подключается через `Catalog`, а позиции — через `SearchableItem`.
Поэтому позднее можно заменить JSONL на PostgreSQL, не меняя поисковый сервис.
REST API и amoCRM должны использовать тот же `SearchResponse`, а не
реализовывать ранжирование повторно.

## Вопросы проверки и тестирования

Полный накопительный реестр всего проекта находится в
`docs/VERIFICATION_REGISTER.md`. В нём фиксируются статусы `VERIFIED`,
`PARTIALLY_VERIFIED`, `NOT_VERIFIED`, `BLOCKED`, способ проверки,
фактический результат и путь к подтверждению.
