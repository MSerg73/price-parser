# Industrial Price Parser 0.7.1rc14

## Изменение RC14 — проверяемое применение LLM

RC14 подключает результаты LLM к основному конвейеру только после
детерминированной проверки источника.

- локальный parser восстанавливает профиль из явного слова в описании, если
  отдельная колонка профиля пуста;
- пример `Титан уголок 4х4 2740мм` разбирается как профиль `УГОЛОК`,
  материал `ТИТАН`, размеры `4 × 4 × 2740`;
- полный снимок исходной строки и заголовков сохраняется в `source_audit.jsonl`;
- LLM для каждого изменяемого поля обязана вернуть точный `evidence` и короткую
  проверяемую причину;
- применяются только поля решения `PROPOSE_CHANGE`, подтверждённые исходным
  текстом или исходной колонкой;
- поиск и итоговый XLSX строятся после reconciliation;
- аудит `before / proposal / after` сохраняется в
  `llm_application_audit.json`;
- генерация переводов, химических символов и синонимов материалов отключена;
  `Титан → titanium / титановый / Ti` не создаётся.

Формат итогового XLSX из десяти колонок не изменён. Материал сохраняется во
внутренней модели, поисковом тексте, комментарии и audit-файле.

## Изменение RC13 — staging-fixture

RC13 исправляет установщик: staging-копия теперь включает каталог `pilot`,
а обязательный файл `pilot/pilot_gold_v0_7_1_rc3.jsonl` проверяется до запуска
полного pytest. Функциональная логика универсального поиска не изменялась.

## Изменение RC12 — исправление QA и двухэтапная установка

- Исправлен устаревший интеграционный тест, который всё ещё ожидал
  преобразование `Квадрат 100` в старый диаметр-ориентированный синтаксис
  `Квадрат ф100`.
- Начиная с RC11 заявка разбирается напрямую как универсальная маска и не
  переписывается в `ф100`.
- Установщик сначала проверяет пакет в отдельной staging-копии проекта, и
  только после успешных compileall, targeted pytest и полного pytest меняет
  рабочую копию.
- При ошибке установщик выводит исходную причину и хвост соответствующих
  stdout/stderr непосредственно в консоль.

## Изменение RC11 — универсальная поисковая маска

Полная заявка разбирается в необязательные поля:

```text
Наименование/профиль | Марка | Размер 1 | Размер 2 | Размер 3
```

Ни одно поле не является обязательным. Примеры:

```text
Квадрат 100
Пруток150х90
Профильная труба 150 х 60
Аноды шариковые Cu-DHP Ø 25 мм
Аноды плоские Cu-DXP 10 x 100 x 1600
БрБ2 ф20
Стол 1000х1500
```

Последний запрос является корректным: при отсутствии совпадений создаётся
пустой поисковый XLSX с заголовками, а отчёт содержит `Позиции не найдены`.

Пробелы, табуляция и неразрывные пробелы не влияют на разбор. Поддерживаются
разделители `x`, `х`, `×`, `*`. Марки, содержащие `Х` и цифры
(`12Х18Н10Т`), сначала сопоставляются с фактическими марками текущей
объединённой таблицы и не разбираются как размеры. Масса упаковки (`25 кг`) не
попадает в размерности.



## Изменение RC10 — поиск без обязательной марки

Полная поисковая заявка больше не требует марку, если пользователь её не
указал. Запрос:

```text
Квадрат 100
```

в RC10 нормализовался в `Квадрат ф100`; начиная с RC11 разбирается напрямую как маска и ищет все позиции профиля `КВАДРАТ`
независимо от марки. Сначала выводится точная сторона `100`, затем ближайшие
размеры. Фактическая марка каждой позиции сохраняется в результате.

Если марка задана, фильтр остаётся обязательным:

```text
Квадрат БрБ2 100
Квадрат ст.20 100
```

Первый запрос фильтрует по `БРБ2`, второй — по числовой марке `20`.

## Изменение RC9 — все спорные строки

В демонстрационном `run-folder` снят общий лимит `25` строк. По умолчанию
обрабатываются **все** REVIEW-кандидаты. Чтобы не отправлять сотни строк одним
огромным запросом, они передаются последовательно пакетами по `10` строк.

Это не ограничение покрытия: при `559` кандидатах будут выполнены `56`
последовательных пакетов (`55 × 10 + 1 × 9`). Итоговый XLSX при обязательном
LLM создаётся только после успешной обработки всех кандидатов.

В отчёте фиксируются:

- полное число кандидатов;
- число переданных и возвращённых строк;
- полнота покрытия;
- размер и количество пакетов;
- суммарные токены, время и оценочная стоимость;
- отдельный `llm_batches.json` с результатом каждого пакета.

Положительный `--llm-max-cases` сохранён только как явный диагностический
параметр. Значение по умолчанию `0` означает «без ограничения».


## Изменение RC8

Для профиля `КВАДРАТ` одиночное число в поисковой заявке теперь трактуется
как сторона квадрата (`Размер 1`):

```text
Квадрат 100
```

Внутри поискового движка запрос безопасно нормализуется в `Квадрат ф100`.
Исходный текст остаётся в имени XLSX и отчёте.

Чтобы числовое значение считалось маркой, его нужно указать явно:

```text
Квадрат ст.20 100
Квадрат марка 20 100
```

Два немаркированных числа не угадываются: такая заявка остаётся без
автоматической перестановки. Исправлен также BOM в `START_DEMO.cmd`, из-за
которого Windows показывал ошибку `я╗┐@echo off`.

## Базовое состояние RC7

`0.7.1rc7` исправляет три дефекта, обнаруженные на четвёртом прайсе:

1. локальная конфигурация `.env` теперь загружается из корня проекта;
2. новый прайс с колонками `Наименование / Вид проката / Диаметр /
   Размеры мм. / Остаток в кг` разбирается по назначению колонок;
3. демонстрационный поиск принимает полную заявку `пруток БрБ2 ф20` и
   показывает точные совпадения перед ближайшими размерами.

Релиз не считается проверенным на live API до фактического запуска на рабочем
компьютере с локальным ключом и получения ненулевого отчёта токенов.

## Запуск демонстрации

1. Положить все прайсы в `C:\AI Test\input`.
2. Проверить наличие `C:\AI Test\price_parser\.env`.
3. Запустить `C:\AI Test\demo_current\START_DEMO.cmd`.
4. Ввести:

```text
пруток БрБ2 ф20
```

При успешном обязательном LLM-вызове создаются:

- `Общая.xlsx` — все позиции всех текущих прайсов;
- `Поиск пруток БрБ2 ф20.xlsx` — точные совпадения, затем ближайший диаметр;
- `processing_report.txt/json` — время, модель, строки LLM и токены.

Если LLM обязателен, но ключ не загружен или API завершился ошибкой, итоговые
XLSX не создаются и запуск завершается со статусом `FAILED_LLM_REQUIRED`.

## Конфигурация `.env`

Файл:

```text
C:\AI Test\price_parser\.env
```

Минимальное содержимое:

```env
OPENAI_API_KEY=...
LLM_MODEL=gpt-5.4-mini
LLM_API_KEY=
```

Правила:

- реальный ключ не включается в ZIP, Git или отчёт;
- `OPENAI_API_KEY` имеет каноническое имя;
- `LLM_API_KEY` — только legacy-совместимость;
- если оба ключа заданы и различаются, выполнение блокируется;
- существующие переменные процесса имеют приоритет над `.env`;
- `.env` поддерживает ASCII/UTF-8/UTF-8 BOM и Windows CRLF.

## Разбор нового прайса

Для таблиц с явными заголовками используются следующие роли:

| Исходная колонка | Итоговое поле |
|---|---|
| Наименование | марка/обозначение и исходный комментарий |
| Вид проката | Профиль |
| Диаметр | Размер 1 |
| Размеры мм. | следующие размеры по правилу профиля |
| Остаток в кг | Наличие |

Пример:

```text
ЭИ868(ХН60ВТ) | круг | ф105 | 360мм | 28.9
```

Результат:

```text
Профиль: КРУГ
Марка: ХН60ВТ
Размер 1: 105
Размер 2: 360
Наличие: 28.9 кг
Комментарий: исходное обозначение ЭИ868
```

Если на вторичном листе заголовок находится ниже первых строк, он определяется
в пределах первых 500 строк. Префикс без заголовка может использовать только
схему, уже подтверждённую явным заголовком в том же workbook.

Если вид проката отсутствует, марка или размер не подставляются в колонку
`Профиль`: строка получает `НЕ УКАЗАН` и направляется в REVIEW.

## Полная заявка и приоритет поиска

Запрос:

```text
пруток БрБ2 ф20
```

Разбирается как:

- профильная группа: `ПРУТОК/КРУГ`;
- нормализованная марка: `БРБ2`;
- целевой диаметр: `20 мм`.

Порядок результата:

1. точное совпадение марки, профильной группы и диаметра;
2. та же марка и профильная группа с ближайшим диаметром;
3. далее — по увеличению абсолютной разницы размера.

В полном поиске исходные позиции не дублируются искусственными alias-строками.

## LLM-контур

Локальный парсер обрабатывает однозначные строки. В LLM передаются все
спорные строки за запуск, последовательно пакетами по `10`. Приоритет получают
строки неизвестного generic-поставщика с сохранёнными исходными колонками.

LLM получает:

- source ID;
- исходное описание;
- текущие профиль, марку и размеры;
- значения распознанных исходных колонок;
- причины REVIEW;
- доменные запреты.

Ответ проходит Pydantic/JSON Schema, проверку количества строк, source ID,
дубликатов, confidence и evidence. Предложения LLM не применяются к основной
таблице автоматически; они сохраняются в `llm_safe` для проверки.

Для Responses API используется `store=false`. Токены берутся только из
фактического поля `usage`; отсутствующие метрики обозначаются как
`не предоставлено`, а не выдумываются.

## Выходная схема XLSX

1. Поставщик
2. Профиль
3. Марка
4. Размер 1
5. Размер 2
6. Размер 3
7. Наличие
8. Цена (₽/кг)
9. Комментарий
10. Источник (файл/строка)

## Ошибки и восстановление

| Симптом | Значение | Действие |
|---|---|---|
| `.env` не найден | Ключ не загружен | Проверить `C:\AI Test\price_parser\.env` |
| `LLM обязателен, но ключ не загружен` | Нет `OPENAI_API_KEY`/`LLM_API_KEY` | Заполнить локальный `.env`, не отправляя ключ |
| Конфликт двух ключей | Обе переменные заданы и различаются | Оставить один проверенный ключ |
| `FAILED_LLM_REQUIRED` | API не подтвердил обязательный live-вызов | Открыть последний `processing_report.txt`; XLSX намеренно не создан |
| `401/403` | Ключ недействителен или нет доступа | Проверить ключ/проект OpenAI |
| `429` | Лимит или временное ограничение | Проверить лимиты и повторить один раз позже |
| Модель не найдена | Неверное значение `LLM_MODEL` или нет доступа | Проверить модель в API-проекте |
| Токены `не предоставлено` | Провайдер не вернул usage | Не считать значением 0 |
| Профиль равен марке (`ЭИ868`, `ХН60ВТ`) | Используется старая версия или схема не распознана | Проверить версию `0.7.1rc7` и источник строки |
| Размеры пустые при заполненных `Диаметр/Размеры мм.` | Используется старая версия или новый дрейф колонок | Проверить `source_columns` в LLM-кандидате и отчёт |
| Поиск принимает только `Круг`/`Пруток` | Старый launcher RC6 | Обновить `START_DEMO.cmd` из RC7 |
| Создан только профильный XLSX | Старый режим запуска | Ввести полную заявку; RC7 создаёт общую и поисковую таблицы |
| `Python проекта не найден` | Нет `.venv` или неверный путь | Проверить `.venv\Scripts\python.exe` |
| Нет файлов `.xls/.xlsx` | Папка `input` пуста | Положить файлы непосредственно в `input` |
| Файл `~$...` пропущен | Временный lock-файл Excel | Закрыть Excel; это штатно |
| XLSX не записывается | Файл открыт/нет прав | Закрыть Excel и проверить права на `demo_current` |
| PowerShell показывает `>>` | Незавершённая команда | Нажать `Ctrl+C`, вставить только команду без `PS ...>` |
| Установщик: `FAILED`, `Rollback: SUCCESS` | QA не прошёл | Предыдущая версия восстановлена; открыть отчёт установщика |

## QA релиза

Локально подтверждено:

- компиляция Python;
- targeted-тесты generic parser, `.env`, full query и mandatory LLM;
- полный pytest;
- обработка четырёх предоставленных прайсов без внешнего ключа;
- контроль строк SupplierDelta `534–540`;
- поиск `пруток БрБ2 ф20`: точные совпадения перед ближайшими.
